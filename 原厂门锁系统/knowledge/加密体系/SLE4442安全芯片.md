### 2.4 IniUsb 实现分析

真正的函数体在 `0x10002410`:

```asm
IniUsb:
  push ebp
  mov  ebp, esp
  sub  esp, 0x40
  mov  ecx, 0x10017950       ; 设备对象实例
  call 0x10001082             ; 调用构造函数 (C++ 对象初始化)
  ; 返回 eax = 对象指针
```

实际逻辑非常薄——只是创建一个 C++ 对象来管理 USB 连接。真正的初始化 (DeviceIoControl 握手) 发生在后续的 `readPorta/readPortb` 调用中。

### 2.5 writePorta 实现分析

```asm
writePorta:
  ; 1. 设置 SEH 异常处理框架
  push -1
  push 0x10004040            ; 异常处理器
  mov  eax, fs:[0]
  push eax
  
  ; 2. 初始化设备句柄为 INVALID_HANDLE_VALUE
  mov  [ecx], 0xffffffff
  
  ; 3. 准备 16 字节输出缓冲区
  push 0x10                  ; 16 字节
  push 0                     ; flags
  push ecx+4                 ; buffer
  
  ; 4. 从传入参数复制 16 字节到缓冲区
  mov  edx, [param1]         ; 源数据指针
  mov  [ecx+4], edx          ; 复制 4x4=16 字节
  
  ; 5. 调用 DeviceIoControl
  push size                  ; nOutBufferSize (0x12)
  push 0                     ; lpOverlapped
  push 0                     ; lpBytesReturned
  push param1                ; hDevice handle
  call [0x100194f4]          ; → DeviceIoControl
  
  ; 6. 检查返回值
  cmp  [ecx], -1
  jne  success
  call [0x100192d0]          ; → GetLastError
  ; 返回错误码
```

**关键发现:** d12.dll 完全是一个薄封装——它只是把参数转成 Windows USB API 调用，没有任何加密或握手逻辑在 DLL 层面。

### 4.3 SLE4442 安全芯片模型

SLE4442 是西门子 (现英飞凌) 的智能卡安全芯片：

```
SLE4442 存储结构:
┌─────────────────────┐
│  主存储器 (256字节)   │ ← 存放门锁卡数据
├─────────────────────┤
│  保护存储器 (32位)    │ ← 每位保护 8 字节区间
├─────────────────────┤
│  安全存储器 (4字节)   │
│  [0] EC  (错误计数器) │ ← 0-7, 出厂=7, 3次错误后=0 → 永久锁死!
│  [1] PSC0            │ ← 可编程安全码 字节0
│  [2] PSC1            │ ← 可编程安全码 字节1
│  [3] PSC2            │ ← 可编程安全码 字节2 (出厂=0xFF)
└─────────────────────┘

安全码校验流程:
1. 验证 PSC: EC≠0 → 比较输入 vs 存储的 PSC
2. 正确: EC=7 (恢复)
3. 错误: EC--; 如果 EC=0 → 永久锁死
4. EC=0 时: 所有写操作被芯片内部硬件拒绝
```

### 4.4 cpu_protocol — CPU 通信协议核心

从反汇编还原的 AVR 通信协议包格式:

```
┌──────┬──────┬──────┬──────┬──────┬─────────┬──────┐
│ 0xAA │ 0x21 │ 0x80 │ Len  │ 0x20 │ Payload │ Chk  │
│ 头   │ 命令 │ ?    │ 长度  │ 偏移 │ 数据    │ 校验 │
└──────┴──────┴──────┴──────┴──────┴─────────┴──────┘

命令包结构:
  [0]    0xAA        ← 固定包头
  [1-2]  命令字       ← 0x21 80 (初始化命令)
  [3]    子命令       ← 0x03 (固定)
  [4]    数据长度     ← 可变
  [5]    0x20        ← 起始偏移
  [6..]  载荷数据
  [尾]   校验和       ← 简单 XOR

响应包:
  [0]    状态字节     ← 0=成功, 非0=错误码
  [5..]  返回数据     ← 如版本信息、序列号等
```

### 4.5 auto_init — 自动初始化流程

```python
def auto_init(port, baud):
    # 1. 初始化 IO 端口
    ic_init(port, baud)
    
    # 2. 探测芯片是否存在
    for retry in range(max_retries):
        result = auto_chk(port)
        if result >= 0:
            break
        sleep_delay()  # 等 2000ms
    
    # 3. 设置波特率
    set_baud(port, detected_baud)
    
    # 4. 尝试 CPU 通信
    result = cpu_protocol(port, command, data)
    if not success:
        return -136  # 0xFFFFFF78
    
    # 5. 重置安全芯片
    ic_exit(port)
    
    return handle  # 成功返回句柄
```

## 五、259 (0x103) 错误码溯源

### 5.1 搜索结论

- **d12.dll**: 不含 0x0103 常量 → 不是 DLL 层硬编码的
- **d12c.dll**: 含 28 处 0x0103 → 分布在数据段中
- **Mwic_32.dll**: 含 28 处 0x0103 → 分布在数据段中

### 5.2 最可能来源

259 = 0x0103。在 SLE4442 语境下:

```
AVR 固件通过 DeviceIoControl 返回 259 的场景:

1. SLE4442 错误计数器 EC=0 (永久锁死)
   → 固件检测到芯片安全存储器[0]=0x00
   → 返回 259 给 PC 端

2. PSC 验证失败 3 次后
   → 芯片内部硬件锁死
   → 固件无法写入
   → 返回 259

3. 或者: AVR ISP 协议状态码 0x03 = 设备忙/被保护
   加上错误类别码 = 0x0103
```

### 5.3 为什么"永久"

SLE4442 芯片的安全机制是**物理不可逆的**：
- EC 计数器存在芯片内部 EEPROM
- 每次错误 PSC 校验永久减少 EC
- EC=0 后，芯片硬件永久拒绝所有写操作
- **唯一的恢复方式：物理替换芯片**

1. **DirectWriteUSB (d12=0 失败时尝试)**
   - 绕过高层的 V9RFL.dll → GuestCard 路径
   - 直接通过 d12.dll DeviceIoControl 发送裸命令

2. **AVR ISP 重编程**
   - 用 USBasp/USBtinyISP 直接连 AVR MCU

3. **SLE4442 芯片替换**
   - 找到板上的 SLE4442 芯片
   - 用新的空白 SLE4442 替换
   - 出厂 PSC=FFFFFF, EC=7
   - 需要重新走一遍初始化流程

4. **d12=1 绕过**
   - 当前已实现 ✅
   - GuestCard 走 d12=1 能正常工作
   - 其他卡类型需要验证

## 八、结论

1. **d12.dll/d12c.dll 不是加密瓶颈** — 它们只是 USB 驱动的薄封装
2. **真正的"保护锁"是 SLE4442 芯片的 PSC/EC 机制** — 这是硬件级别的安全芯片
3. **259 返回值来源** — 最可能是 AVR 固件在检测到 SLE4442 EC=0 后通过 DeviceIoControl 返回的状态码
4. **没有"握手加密"** — 只有 SLE4442 的标准 PSC 验证机制 + DES 加密的通信数据
5. **破解方向** — DirectWriteUSB 裸命令 + AVR ISP 重编程固件是两个最有希望的方向，但都需要发卡器硬件在手里
6. **V9RFL.dll 层没有独立加密** — 所有加密要么在 Mwic_32.dll (DES)，要么在 SLE4442 芯片内部
