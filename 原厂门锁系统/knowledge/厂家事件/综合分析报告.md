### 3.2 关键字段

| 字段 | 含义 | 长度 | 说明 |
|:-----|:-----|:-----|:------|
| `PCID` | PC Identifier (硬件指纹) | 12 hex | 由主板信息 (MAC/硬盘序列号) 生成 |

| LD | 值 | 来源 | 场景 |
|:---|:----|:------|:------|
| LD-A | `EFF09A7DB6F1E66D375D5DECF7046F90` | N022B_5EE69782.TXT | 旧系统被锁时 |

**关键发现: 全部 4 个 LD 完全不同。验证 LD 不是固定值生成，而是厂家每次给的新授权码。**

### 3.4 锁机判定逻辑 (逆向确认)

CardLock.exe 启动时检查:
```
1. 读取 System.ini → PCID, SN, LD, LDO
2. 用 PCID + SN 验证 LD 签名
3. 若 LD 无效 → 弹出 "N9101: 请注册" → 锁机
4. 若 LDO < 当前日期 → 弹出 "E0073: 授权已过期" → 锁机
5. 若 PCID 与硬件不符 → 弹出 "PCID 不匹配" → 锁机
6. 另外有 ~131 个外围检查点 (cmp byte [eax], 0) 验证授权状态

锁机后果:
- 发卡功能禁用
- 界面锁定
- 所有操作被阻止
```

### 4.3 文件名 = 密钥体系

你提出的核心洞见: **文件名本身就是密钥体系的成员**

| 文件名 | 含义 |
|:-------|:------|
| `CARDLOCK-N8.9.1.EXE` | N = Network 版, 8.9.1 = 版本号, 授权给特定客户的密钥 |
| `CARDLOCK-N02_D.EXE` | N02 = 客户编号 02, D = 版本 D, 另一个授权版本 |
| `REPAIRACCESS.EXE` | 同命名体系的第三个成员, 不是普通修复工具 |

**三层密钥系统:**
1. **命名密钥 (文件名):** CARDLOCK-N8.9.1, CARDLOCK-N02_D — 标识授权版本和客户
2. **注册密钥 (LD):** System.ini 中的 `LD=xxxxxxxx` — 实际授权串
3. **校验链 (PCID + SN + LD):** 三者绑定验证 — 缺一不可

### 4.4 Prefetch 中 N 系列 EXE 的引用文件 (非系统)

两个 N 系列 EXE 都引用:
- `System.ini` — 修改 LD/LDO
- `CardLock.MDB` — 同步数据库授权记录
- `D12C.DLL` — 验证 LD 合法性的 DLL

## 7. Binary 二进制补丁记录

### 7.1 补丁演进

全部补丁均针对 `CardLock.exe` (版本: DC 05/17/2011)。

| 补丁 | 文件 | 修改字节数 | 目标 | 方法 |
|:-----|:-----|:----------|:-----|:------|
| F2 | patch_reg.py / patch_reg_v2.py | ~34 bytes | 注册检查点 | JMP 跳表: 将条件跳转 (JZ/JNZ) 改为 JMP 跳过注册检查 |
| F4 | CardLock_F4 | 34 bytes | 类似 F2 | 进一步覆盖注册判断 |
| F5 | apply_f5_2_correct.py | 6 bytes | N9101 + E0073 | 74->EB (JZ->JMP), 90->EB 2A 跳过 "请注册" 和 "过期" 弹窗 |
| F6 | apply_f6_patch.py | 4 bytes | 131 个外围检查点 | 指针重定向: 0x76FABC 的指针从 BSS 段 (运行时清零) 改为 CODE 段 (0x401000, 始终非零) |

### 7.2 F5 补丁详解

修改 3 处:
```
1. 0x33EAD8-0x33EAD9: 90 90 (NOP) -> EB 2A (JMP跳过N9101显示)
   原: NOP NOP (之前的错误修复会落到N9101显示)
   改: JMP +0x2A (直接跳过整个 "请注册" 路径 + N9101 弹窗)
   
2. 0x33E32C: 74 -> EB (JZ->JMP, 跳过 E0073-1)
   原: JZ (条件跳转, 不满足条件时显示弹窗)
   改: JMP (无条件跳转, 永远跳过弹窗)
   
3. 0x33E557: 74 -> EB (JZ->JMP, 跳过 E0073-2)
   原: JZ (第二个过期检查)
   改: JMP (无条件跳过)
```

### 7.3 F6 补丁详解

```
原理: CardLock.exe 有 131 个 "cmp byte [eax], 0" 检查点
      这些检查点通过一个 BSS 段的指针 (0x0077AF8C) 寻址
      运行时该指针指向的内存被清零 → 所有检查点都通过 → 报未注册

修改: 改变指针值
  原指针: 0x0077AF8C (BSS段, 运行时清零)
  新指针: 0x00401000 (CODE段, 固定非零字节 = 0x04)
  
  位置: 文件偏移 0x36E6BC (对应 VA 0x76FABC)
  修改: [旧值4字节] -> [新值4字节]

效果: 所有 131 个检查点看到 [eax] 的字节是 0x04 ≠ 0 → 被认为已注册
安全: 这些检查点都是只读的 (cmp byte [eax], 0), 没有写操作
```

### 7.4 补丁效果总结

```
检查点类型         数量    处理方式
────────────────────────────────
注册弹窗 (N9101)     ~3    F5: JMP 跳过
过期弹窗 (E0073)     ~2    F5: JZ->JMP
核心注册验证         ~34   F2/F4: JMP 跳表
外围授权检查         131   F6: 指针重定向 (4字节改一个 dword)
───────────────────────────────────
总计绕过           ~170 个检查点 (共修改 ~44 字节)
```

```

    │
    └── Mwic_32.dll (126KB, 244 导出函数) — 安全 + 加密层
          ├── SLE4442 安全芯片操作 (ic_init, csc_4404, rsc_4404...)
          ├── DES 加密/解密 (DES_Encrypt, DES_Decrypt)
          ├── AVR CPU 协议 (cpu_protocol, cpu_comres)
          └── 自动初始化 (auto_init, auto_chk)
```

### 8.2 关键结论

- **d12.dll / d12c.dll 不是加密瓶颈** — 只是 USB 驱动的薄封装
- **真正的"保护锁"是 SLE4442 芯片的 PSC/EC 机制** — 硬件安全芯片
- **259 错误码** — 最可能是 AVR 固件检测到 SLE4442 EC=0 后返回的状态码
- **V9RFL.dll 层没有独立加密** — 所有加密在 Mwic_32.dll (DES) 或 SLE4442 芯片内部

### 8.3 SLE4442 安全芯片

```
SLE4442 存储结构:
┌─────────────────────┐
│  主存储器 (256字节)   │ ← 门锁卡数据
├─────────────────────┤
│  保护存储器 (32位)    │ ← 每位保护 8 字节区间
├─────────────────────┤
│  安全存储器 (4字节)   │
│  [0] EC  (错误计数器) │ ← 出厂=7, 3次错误后=0 → 永久锁死
│  [1] PSC0            │ ← 可编程安全码 字节0
│  [2] PSC1            │ ← 可编程安全码 字节1
│  [3] PSC2            │ ← 可编程安全码 字节2 (出厂=0xFF)
└─────────────────────┘

EC=0 后: 芯片硬件永久拒绝所有写操作, 唯一恢复方式 = 物理更换芯片
```
