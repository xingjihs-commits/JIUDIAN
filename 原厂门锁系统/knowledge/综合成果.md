## 二、六层加密体系

### 加密全景

```
系统启动 ──→ System.ini (PCID/dlsCoID) ──→ 数据库连接 ──→ 发卡操作 ──→ MIFARE 卡数据
                │                              │              │              │
                ▼                              ▼              ▼              ▼
          第1层: 算法密钥              第4/5/6层: DB加密   第2层: LockNo  第3层: CardData
          (AES keys 从这里来)           (MySQL/AES)         (BCD 变换)     (AES-256 + 偏移)
```

### 各层详情

| 层 | 名称 | 算法 | 说明 |
|:--:|:-----|:-----|:------|
| 1 | 密钥派生 | PCID → SHA256 → AES Key | System.ini 的 PCID 字段被哈希后做 AES 密钥 |
| 2 | LockNo 变换 | BCD nibble 位交换 | 发卡前 LockNo 做位变换，门锁还原 |
| 3 | CardData 加密 | AES-256-CBC | MIFARE 卡 16 字节 payload 的 Body 加密 |
| 4 | MySQL 连接 | 用户名/密码/Database | 数据库连接凭据 |
| 5 | 表级 AES | AES-128-ECB | 部分敏感字段列级加密 |
| 6 | DCPcrypt | Delphi DCPcrypt 库 | CBC + 偏移量模式 |

### 关键 DLL

| DLL | 导出数 | 作用 |
|:----|:------:|:-----|
| **V9RFL.dll** | 77 个 | 门锁业务编排层（CardLock.exe 只用 8 个） |
| **d12.dll / d12c.dll** | 4 个 | USB 通信层（IniUsbPro, readPortPro, writePortPro, CloseHandles） |
| **Mwic_32.dll** | 244 个 | 安全芯片 + AVR 协议 + DES 加密（126KB） |

### V9RFL.dll 8 个关键导出

| 函数 | 用途 |
|:-----|:------|
| `initializeUSB` | 打开 USB 连接（d12=0 完整模式, d12=1 轻量模式） |
| `CloseUSB` | 关闭 USB 连接 |
| `ReadCard` | 读卡到缓冲区 |
| `WriteCard` | 写缓冲区到卡 |
| `IniCard` | 初始化卡片（发新卡前调用） |
| `CardErase` | 擦除卡片 |
| `Buzzer` | 蜂鸣器 |
| `proTest` | 自检 |

### SLE4442 安全芯片

- 发卡器内部硬件安全芯片
- PSC（安全码出厂 = FFFFFF）
- EC（错误计数器，最大值 7）
- EC=0 后 **永久锁死写操作**，不可软件恢复
- `initializeUSB(d12=0)` 259 返回码来源于 AVR 固件检测到 SLE4442 异常

## 三、卡数据体系（全卡型结构+校验）

### 通用 Payload 结构（16 字节）

```
偏移  长度  字段      说明
0~3   4     Magic     固定 C92B20B7
4~5   2     Site      酒店码（dlsCoID 低16位）
6~7   2     LockNo    锁号 / 通配
8     1     Salt      盐值（通常 00）
9     1     Type      高4bits=卡类型, 低4bits=序号
10~13 4     Body      类型相关数据
14~15 2     Checksum  校验和
```

### 各卡型速查

| 卡型 | Type 高4bits | 结构特征 | 样本数 |
|:-----|:------------:|:---------|:------:|
| AuthCard（授权卡） | 0x0 | LockNo 固定 `809C`，最后6字节=3字节 Token×2 | 14 |
| GuestCard（客人卡） | 0x6 | Body 含 BCD 时间戳（年+月日） | 多 |
| MasterCard（总卡） | 0xB | byte[15] 固定 `FB` | 18 |
| BuildingCard（楼栋） | 0xC | byte[15] 固定 `FB` | 1 |
| FloorCard（楼层） | 0xD | byte[15] 固定 `FB` | 1 |
| EmergencyCard（应急） | 0xA | **无样本，需现场采集** | 0 |
| IniCard（初始化卡） | 0x1 | | |

### 校验和

**系统卡（Master/Building/Floor）：**
```
byte[15] = 0xFB  恒成立（18+张样本全部一致）
byte[14] = 每张不同，算法待定
```

**授权卡（AuthCard）：**
```
byte[12:15] = byte[10:12] 复制一次（token×2）
无独立校验和
```

**客人卡（GuestCard）：**
```
当前代码 sum(payload[:14]) & 0xFF → ❌ 与实际不匹配
正确算法未知，需更多样本
```

### 时间戳编码

**GuestCard：**
```
byte[12] = yy（BCD 年）
byte[13] = mmmmdddd（BCD 月日）
byte[14:15] = 校验和 + 00
```

**系统卡：** Body[10:14] 含时间戳，格式待更多样本分析

### AuthCard Token 公式

Token = f(Site, Seq, Timestamp) — 不能从静态样本推导。
短期方案：已知 {seq: token} 查找表。
长期方案：algo_study 矩阵现场采集 → 反推算法。

## 四、授权机制（LD/LDO/注册）

### System.ini 关键字段

| 字段 | 长度 | 说明 |
|:-----|:----:|:------|
| `PCID` | 12 hex | 电脑硬件指纹（CPU+主板哈希） |
| `SN` | 可变 | 发卡器序列号 |
| `dlsCoID` | 数字 | 经销商 ID（本酒店 = 2826423） |
| `LD` | 32 hex | **授权码** — 核心锁机字段 |
| `LDO` | 日期 | **授权到期日** |

### LD 版本演化

| 时间 | LD（前16字节） | LDO | 来源 |
|:----:|:---------------|:----:|:-----|
| 初始 | `E5BC2456664279810E4454037D207DDC` | — | 总卡注册 |
| 06/03 被锁 | `EFF09A7DB6F1E66D375D5DECF7046F90` | 2026/04/16 | 厂家注入 |
| 06/04 解锁 | `47B61A8933E299F4...` | 2026/05/15 | N8.9.1 |
| 06/04 再注 | `A3F8C12D5E6B90C7...` | 2026/06/01 | N02_d |

### d12 锁定机制

| 模式 | 说明 | 返回值 |
|:-----|:------|:------:|
| `initializeUSB(d12=0)` | 完整模式 USB 握手 | 259（发卡器固件保护锁定） |
| `initializeUSB(d12=1)` | 轻量模式 USB 句柄 | 0（成功） |

**关键：** d12=1 时 `ReadCard`、`WriteCard`、`IniCard` 全部正常工作。
**Solid PMS 发卡走 d12=1，完全不受 259 锁影响。**

### 5.3 PATH 1 — 纯注册码校验（006864DD ~ 0068664F）

> 走这条路的条件是 **GroupBox3.byte[57] ≠ 0**  
> 当 `edtRegistor1.Text` 有内容（已输入了注册码）时触发

#### 第一步：读编辑框内容

| VA | 汇编 | 注释 |
|:--:|:------|:------|
| `006864DD` | `lea edx, [ebp-24]` | 目标字符串 |
| `006864E0` | `mov eax, [ebp-4]` | Self |
| `006864E3` | `mov eax, [eax+3F0]` | **Self.Memo2** (编辑框/备注框, 偏移 3F0) |
| `006864E9` | `call TPanel.GetCaption` | **取 Memo2.Text** → 即 `edtRegistor1.Text` |

#### 第二步：取密钥 'aa02bb02CC02DD02'

| VA | 汇编 | 注释 |
|:--:|:------|:------|
| `006864EE` | `mov eax, [006B7408]` | 全局变量指针 (^gvar_006B9CE4) |
| `006864F3` | `mov edx, 68755C` | **硬编码字符串: `'aa02bb02CC02DD02'`** |
| `006864F8` | `call 004045B0` | 字符串赋值操作 |
| `006864FD-00686526` | 逐字节拷贝 | 将密钥复制到 [ebp-65] 局部缓冲区 |

#### 第三步：XOR 解密第一段 (from Memo2[1..32])

```
a_hex(Memo2[1..32]) → hex 字符串转二进制
     │
     ▼
XOR with 'aa02bb02CC02DD02'
     │
     ▼
hex_a() → 转回 hex 字符串 → 存到 [ebp-8]
     │
     ▼
拼接 "第1段[1..32]" + "第2段[33..64]" → 存回 [ebp-8]
```

汇编细节：

| VA | 操作 |
|:--:|:------|
| `00686528-00686556` | 取 Memo2[1..32], a_hex 转二进制 |
| `0068655B-0068656A` | call `0068F1B0` (XOR 解密, key=aa02bb02CC02DD02) |
| `0068656F-0068657F` | hex_a 转回 hex 字符串 |
| `00686584-0068664A` | 拼接完整64字符, 存 [ebp-8] |

重复同样的步骤对 Memo2[33..64] 做第二次 XOR：

| VA | 操作 |
|:--:|:------|
| `006865B3-0068660A` | 取 Memo2[33..64], a_hex → XOR → hex_a |
| `0068660F-0068664A` | 拼接 → 存 [ebp-8] |

**这两段 XOR 解密就是所谓的 "LD 解码"——把用户输入的注册码第1步解开。**

#### 第四步：跳过 PATH 2

| VA | 汇编 | 注释 |
|:--:|:------|:------|
| `0068664F` | `jmp 006866B7` | **跳过 PATH 2，直接合并到公共路径** |

> ```

### 5.9 注册码验证完整断言

成功的注册码必须满足 **全部** 以下条件：

```
1. 原始长度 ≥ 128 hex 字符 (64 字节二进制)
2. 用密钥1(aa02bb02CC02DD02) XOR 解密后 → [ebp-8]
3. 用密钥2(LHPV9XL2800867RJ) XOR 解密后 → [ebp-0C]
4. [ebp-0C] 长度 == 64
5. [ebp-0C][1..2] hex == "20"          ← 固定前缀
6. [ebp-0C][27..50] hex == SN          ← 发卡器序列号 (24 hex)
7. [ebp-0C][51..54] hex == dlsCoID     ← 酒店码 (4 hex)
8. [ebp-0C][40] == "0"                 ← 注册成功标记
9. [ebp-8] 长度 == 80h (128字节)        ← 扩展数据
```

**任何一条不满足 → 弹 E0074 + 启动 DoNotClickMe.exe 自动更新 + 终止进程。**

### 5.10 注册成功的写回操作

当全部校验通过后：

1. **关 Timer1、Timer2、Timer3** — 永不再校验 (一次性通过)
2. 写入 `'000000'` 到某注册标志位
3. 把 [ebp-8] 按 4 段 (每段 32 hex) 分别用不同的密钥 XOR 解密
4. 计算两个 16-bit 校验值 (XOR 0B418h / 6302h)
5. **创建 TIniFile → 写 System.ini:**
   - `[SYSTEM]` 段 → `SN = <解码出的 SN>`
6. call `sub_006813B4` → 生成注册数据
7. call `sub_00693E74` (V9RFL.dll 另一导出函数) → 写入读卡器
8. 解析解码数据 → 算出日期 → **写 LDO = 新到期日到 System.ini**
9. 界面恢复 (Timer 保持关闭)

  "CardLock" = 固定前缀
  "N" = 标记为授权注入器
  "8.9.1" / "02_d" = 密钥参数
      │
      ▼
修改 System.ini:
  ├── LD  ← 用文件名派生的密钥产生新的32字节授权码
  └── LDO ← 设置新的到期日期
      │
      ▼
bat 自删除（只留 Prefetch 痕迹）
      │
      ▼
CardLock.exe 重新校验 → LD匹配 → 正常启动
```

### 关键发现

- **文件名即密钥**：N 后缀（如 `N8.9.1`、`N02_d`）是生成 LD 的种子
- **自毁执行**：执行后 bat 脚本自动删除，只剩 Prefetch 文件
- **不修改其他文件**：只动 System.ini 的 LD/LDO
- **可远程投递**：通过 UltraViewer 文件传输功能

## 十二、吃掉战略 — 当前实际架构

> **核心理念不变：不是打补丁——是绕过 EXE 直接调 DLL，让厂家彻底与你无关。**
> 但实际实现走的不是原先设计的 FrontDeskService.exe 独立进程路线，
> 而是**嵌入式桥接方案**——发卡逻辑直接嵌入 PMS 主进程。

### 当前实际架构

```
PMS 主进程 (Solid.exe, 64-bit)
    │
    ├── lock_issue_service.py          ← 发卡统一入口
    │       │
    │       ├── prousb_v9.py           ← V9 发卡适配器
    │       │       │
    │       │       ├── 32-bit 桥子进程 (rfl_bridge_32.exe)
    │       │       │       └── V9RFL.dll (32-bit 原生 DLL)
    │       │       │
    │       │       ├── BrandPayloadFactory + profile JSON
    │       │       │       └── DirectWriteUSB 裸写（系统卡用）
    │       │       │
    │       │       └── 30s 心跳保活 timer
    │       │
    │       ├── GenericLockAdapter      ← 通用适配器（profile 驱动）
    │       └── CardLockAutoAdapter     ← pywinauto 寄生 fallback
    │
    └── lock_adapters/
            ├── profiles/              ← 10 个品牌 profile JSON
            ├── __init__.py            ← 品牌适配器注册（10 个内置 + 通用）
            └── bridge_client.py       ← 32-bit 桥通信客户端
```

### 与最初方案的关键差异

| 维度 | 最初设计 | 实际实现 |
|:-----|:---------|:---------|
| 独立 EXE | FrontDeskService.exe | **无**，发卡嵌入 PMS 主进程 |
| 32 位桥 | 桥 + FrontDesk 都用 32-bit | 桥是独立 32 位子进程，PMS 是 64 位 |
| 品牌配置 | YAML 格式 | JSON 格式（`lock_adapters/profile/profiles/*.json`） |
| 品牌检测 | 扫描目录匹配 | `detect.files` + `detect.ini` 匹配 + 适配器 `match()` 方法 |
| 系统卡写卡 | 底层 DirectWriteUSB | DirectWriteUSB + `BrandPayloadFactory` 构造 payload |
| 额外功能 | 补丁扫描器 + 自动部署 | **未实现** |
